# 🧠 UrbanFlow Architecture & ML Pipeline

This document details the technical architecture of the UrbanFlow Command Center, specifically how the raw Jupyter Notebook data science was transformed into a production-grade Decision Intelligence Platform.

## 🏗️ High-Level System Architecture

The platform operates on a decoupled client-server architecture:

1.  **Frontend (Next.js React App):** Handles state management, UI rendering, map visualisations (Leaflet), and animations. Runs on Node.js.
2.  **Backend (FastAPI Python Server):** Exposes REST endpoints (`/api/*`). Serves as the orchestrator for the ML pipeline and mathematical engines.
3.  **ML Engine (`ml_engine.py`):** An in-memory, stateful module initialized on server startup. It loads data, trains the models, and holds the `Predictor` and `Explainer` objects in memory for low-latency inference.

## 🔬 The Machine Learning Pipeline

The core congestion prediction is powered by a custom ensemble approach.

### 1. Feature Engineering
Raw data from `dataset/*.csv` is ingested and processed:
*   **Temporal Features:** Start times are converted to `hour`, `day_of_week`, `is_weekend`, and `is_peak_hour` binary flags.
*   **Spatial Clustering (K-Means):** Latitude and Longitude are fed into a Scikit-Learn `KMeans` model (k=20) to automatically identify and group geographic "Hotspots".
*   **NLP Text Vectorization (TF-IDF):** Unstructured text from the `description` field is processed using `TfidfVectorizer` (max_features=50) to extract hidden signals (e.g., words like "VIP", "Protest", "Flooded").
*   **Target Variable:** The "Impact Score" is a derived metric combining `Duration * Priority * Road Closure Multiplier`, normalized to a 1.0 - 10.0 scale using logarithmic scaling.

### 2. Model Ensemble Strategy
Because `xgboost 2.x` and newer versions of `scikit-learn` have differing validation requirements for `VotingRegressor`, UrbanFlow utilizes a **Custom Averaging Ensemble**:
*   **Model A: XGBoost Regressor:** Tuned via `RandomizedSearchCV` to optimize for non-linear relationships and interactions between temporal/spatial features.
*   **Model B: Random Forest Regressor:** Excellent at handling the sparse matrices generated by the TF-IDF vectorizer and one-hot encoding.
*   **Ensemble:** The final prediction is the mean of both models, ensuring high stability and reducing variance.

### 3. Explainable AI (SHAP)
Predictions are fed into a `shap.TreeExplainer`. The engine extracts the top contributing features (e.g., `is_peak_hour = +1.5 impact`) and translates them into a human-readable narrative string for the UI.

## ⚙️ The Operational Engines

The true value of UrbanFlow lies in taking the `score` generated by the ML pipeline and passing it through a series of operational engines.

### 1. Digital Twin Simulation (`digital_twin.py`)
Uses physics and civil engineering approximations to model traffic realities:
*   **Affected Radius:** Calculated using crowd density formulas based on attendance ($Radius \propto \sqrt{Attendance}$).
*   **Delay Estimation:** Uses a modified Bureau of Public Roads (BPR) function: $Delay = T_0 \times (1 + \alpha (\frac{V}{C})^\beta)$, where $V/C$ is the Volume-to-Capacity ratio derived from the ML impact score.

### 2. Resource Optimizer (`resource_optimizer.py`)
Uses **SciPy Linear Programming (`scipy.optimize.linprog`)** to solve the resource allocation problem.
*   **Objective Function:** Minimize total deployment cost.
*   **Constraints:**
    *   Total officers deployed $\ge$ required coverage based on crowd size.
    *   Vehicles deployed $\ge$ required patrol density based on affected radius.
    *   Maximum available city resources (e.g., max 500 officers).
*   **Output:** The mathematically optimal number of officers, vehicles, and barricades.

### 3. Economic Engine (`economic_engine.py`)
Translates abstract numbers into business value:
*   Calculates total delay hours (Number of affected vehicles $\times$ Avg delay).
*   Applies standard economic formulas: Value of Time (VoT) and Fuel Consumption rates.
*   Outputs the Rupee cost of the congestion and the Kg of $CO_2$ emitted.
*   Compares the "Do Nothing" scenario against the "Optimized Deployment" scenario to prove ROI.

### 4. Diversion Engine (`diversion_engine.py`)
Uses **NetworkX** to build an in-memory graph of the city's road network.
*   Nodes represent intersections; Edges represent road segments.
*   When an event occurs, the engine dynamically modifies edge weights (simulating heavy traffic) within the `affected_radius`.
*   Runs **Dijkstra's Shortest Path algorithm** to find primary, secondary, and emergency routing alternatives around the congestion zone.
